Проект разрабатывался в рамках учебного задания, однако в процессе проектирования архитектуры я принял решение отойти от изначального ТЗ. 

---

Изначальное тз предполагало: передачу файлов через backend (MultipartFile) и хранение путей к файлам в виде строк (`path = "folder/subfolder/file.txt"`).

### 1. Древовидная иерархия (Adjacency List) вместо хранения строковых `path`
**Как в тз:** Предлагалось хранить пути как строки. При переименовании или перемещении папки обновлялись бы пути всех вложенных файлов.

**Моя реализация:** 
* Я использовал паттерн **Adjacency List**: у каждой сущности `Resource` есть `parent_id` (ссылка на родительскую папку). 
* Теперь перемещение папки любого размера это операция **O(1)**, обновляется ровно одна строка.
* Иерархия защищена от зацикливания, папку нельзя перенести папку саму в себя.

### 2. Прямая загрузка (Presigned URLs) вместо проксирования файлов на бэкенде (MultipartFile)
**Как в ТЗ:** Клиент шлет файл на Backend, Backend шлет его в MinIO.

**Моя реализация:** 
* Я применил паттерн **Valet Key**. Бэкенд генерирует криптографическую временную ссылку (MinIO Presigned URL) и отдает её клиенту.
* Клиент загружает файл напрямую в S3, в обход Java-бэкенда. Нагрузка на сервер нулевая.

### 3. REST API: UUID вместо путей
Вместо эндпоинтов вида `GET /resource?path=folder/`. Взаимодействие происходит через уникальные идентификаторы ресурсов: `GET /resources/{uuid}`.

## Стек технологий
* **Бэкэнд:** Java 21, Spring Boot 3, Spring Security, Spring Session
* **База данных:** PostgreSQL, Redis
* **Миграции:** Liquibase
* **S3 хранилище:** MinIO + Minio Java SDK
* **Инфраструктура:** Docker, Docker Compose

## Инструкция по запуску

Проект полностью контейнеризирован. Для запуска вам потребуется только **Docker** и **Docker Compose**.

Проект состоит из двух частей (бэкенд и фронтенд). Склонируйте их в одну рабочую директорию:

### Клонируем Backend
```bash
https://github.com/ramaoao/Cloud-Storage
```

### Клонируем Frontend
```bash
https://github.com/ramaoao/cloud-storage-frontend
```

Перед запуском убедитесь, что в файле `docker-compose.yaml` корректно указаны пути к директориям бэкенда и фронтенда в секциях `build: context`.

### Затем в корне проекта выполните:
```bash
docker-compose up -d --build
```

После завершения сборки и запуска контейнеров, сервисы будут доступны по следующим адресам:

* Frontend: `http://localhost`
* MinIO: `http://localhost:9001` (нужно создать бакет под названием `user-files`)
